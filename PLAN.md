# Implementation Plan: react-fastify-gcp-skel

## Overview and Objectives

Build a reusable monorepo skeleton/template for full-stack React + Fastify applications deployed to GCP. The project replaces the prior `react-fastify-skel` (which used npm workspaces, React Router, and hand-rolled OAuth with Arctic/Jose) with a modernized stack: pnpm workspaces, Turborepo, TanStack Router (file-based), TanStack Query, Shadcn/ui, and Better Auth (replacing the manual Arctic/Jose auth layer).

**Key differences from the reference `react-fastify-skel`:**

| Aspect | Old (react-fastify-skel) | New (react-fastify-gcp-skel) |
|---|---|---|
| Package manager | npm | pnpm |
| Routing | React Router 7 | TanStack Router (file-based) |
| Server state | Manual fetch | TanStack Query |
| Components | Raw HTML/CSS | Shadcn/ui |
| Auth | Arctic + Jose (hand-rolled OAuth) | Better Auth (library handles OAuth, sessions, DB) |
| Auth DB schema | Manual users + oauth_accounts tables | Better Auth managed tables (user, session, account, verification) |

### Step 3.5: Better Auth configuration

**File: `/backend/src/lib/auth.ts`**
```ts
import { betterAuth } from "better-auth";
import { drizzleAdapter } from "better-auth/adapters/drizzle";
import type { DbClient } from "../plugins/db.js";

/**
 * Creates the Better Auth instance.
 * Called after the DB plugin decorates fastify.db.
 */
export function createAuth(db: DbClient) {
  return betterAuth({
    baseURL: process.env.BETTER_AUTH_URL || "http://localhost:3001",
    basePath: "/api/auth",
    database: drizzleAdapter(db, {
      provider: "pg",
    }),
    socialProviders: {
      google: {
        clientId: process.env.GOOGLE_CLIENT_ID as string,
        clientSecret: process.env.GOOGLE_CLIENT_SECRET as string,
      },
      apple: {
        clientId: process.env.APPLE_CLIENT_ID as string,
        clientSecret: process.env.APPLE_CLIENT_SECRET as string,
      },
    },
    trustedOrigins: [
      process.env.FRONTEND_URL || "http://localhost:5173",
    ],
  });
}

export type Auth = ReturnType<typeof createAuth>;
```

**File: `/backend/src/plugins/auth.ts`**

This is the critical plugin that integrates Better Auth with Fastify. Better Auth exposes a standard Web API handler (`auth.handler`). We wrap it in a Fastify route.

```ts
import type { FastifyInstance, FastifyReply, FastifyRequest } from "fastify";
import fp from "fastify-plugin";
import { createAuth, type Auth } from "../lib/auth.js";

export default fp(async (fastify: FastifyInstance) => {
  if (!fastify.db) {
    fastify.log.warn("Database not available — skipping auth plugin");
    return;
  }

  const auth = createAuth(fastify.db);
  fastify.decorate("auth", auth);

  // Mount Better Auth handler at /api/auth/*
  fastify.all("/api/auth/*", async (request: FastifyRequest, reply: FastifyReply) => {
    // Convert Fastify request to Web Request
    const url = `${request.protocol}://${request.hostname}${request.url}`;
    const headers = new Headers();
    for (const [key, value] of Object.entries(request.headers)) {
      if (value) headers.set(key, Array.isArray(value) ? value.join(", ") : value);
    }

    const webRequest = new Request(url, {
      method: request.method,
      headers,
      body: request.method !== "GET" && request.method !== "HEAD"
        ? JSON.stringify(request.body)
        : undefined,
    });

    const response = await auth.handler(webRequest);

    // Convert Web Response back to Fastify reply
    reply.status(response.status);
    response.headers.forEach((value, key) => {
      reply.header(key, value);
    });

    const body = await response.text();
    return reply.send(body);
  });
});

declare module "fastify" {
  export interface FastifyInstance {
    auth: Auth;
  }
}
```

> **Note on plugin load order**: `@fastify/autoload` loads plugins alphabetically. Since `auth.ts` needs `db`, and `cors.ts` / `db.ts` both come before `auth.ts` alphabetically, this works. However, if naming changes, consider prefixing filenames with numbers (`01-db.ts`, `02-auth.ts`) or using explicit registration order in `app.ts`.

## Phase 8: Better Auth Schema Generation

After the backend is set up and dependencies installed:

1. Run `npx @better-auth/cli generate` from the backend directory
   - This generates a Drizzle migration for Better Auth's tables: `user`, `session`, `account`, `verification`
   - The migration goes into `backend/drizzle/`
2. Alternatively, export the schema with `npx @better-auth/cli generate --output backend/src/db/auth-schema.ts` and add it to the drizzle config
3. Run `pnpm --filter backend run db:migrate` to apply migrations locally

Better Auth's managed tables:
- **user**: id, name, email, emailVerified, image, createdAt, updatedAt
- **session**: id, expiresAt, token, createdAt, updatedAt, ipAddress, userAgent, userId
- **account**: id, accountId, providerId, userId, accessToken, refreshToken, idToken, accessTokenExpiresAt, refreshTokenExpiresAt, scope, password, createdAt, updatedAt
- **verification**: id, identifier, value, expiresAt, createdAt, updatedAt

## Dependencies and Risks

### Risks
1. **Better Auth + Fastify integration**: Better Auth is primarily designed for frameworks that use Web API Request/Response (like Next.js, SvelteKit). Fastify uses its own request/response objects. The auth plugin must convert between them. Better Auth does document a `toNodeHandler` utility for Node.js frameworks — the implementer should check if Better Auth provides a `toNodeHandler` and use that instead of manual conversion if available.

2. **Better Auth + Drizzle adapter**: The Drizzle adapter requires the Drizzle instance to be passed to Better Auth. Since Fastify's db plugin creates the Drizzle instance during plugin registration, the auth instance must be created after the db plugin loads. The autoload alphabetical ordering (`auth.ts` after `db.ts`) should handle this, but the implementer should verify.

3. **TanStack Router file-based routing**: The route tree is auto-generated by the Vite plugin. The `routeTree.gen.ts` file must be gitignored and will be regenerated on every `pnpm dev` or `pnpm build`. The implementer should ensure it generates correctly before proceeding.

4. **pnpm + Docker**: pnpm in Docker requires `corepack enable`. The Dockerfiles must enable corepack. Alternatively, the implementer can use `pnpm deploy` for creating a pruned production bundle, which may simplify Dockerfiles.

5. **Shadcn/ui CLI in monorepo**: The `shadcn` CLI must be run from the frontend directory, not the root. It should detect the `components.json` and `vite.config.ts` correctly.

### Mitigations
- If Better Auth's Fastify integration proves problematic, fall back to using Better Auth's `toNodeHandler` which converts it to a standard Node.js `(req, res)` handler, then use Fastify's `fastify.route` with a raw handler.
- If the Drizzle adapter has issues with plugin timing, create the auth instance lazily (on first request) rather than at plugin registration time.
- Run `pnpm dev` early in Phase 4 to verify the TanStack Router Vite plugin generates the route tree before building out all routes.
